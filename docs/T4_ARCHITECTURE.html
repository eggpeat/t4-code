Status

This document defines the accepted architecture for native local use, managed Hub use, and optional Hub-registered workstation execution. T4 Local remains a first-class native product; users do not need a Linux VM or managed stack to operate OMP on the same macOS or Linux computer.

The Flutter client is the product baseline. The current Host evolves into T4 Local Service, while T4 Hub owns distributed managed sessions. Local Service, managed runtimes, and Workstation Runners converge on one T4-owned adapter for pinned official OMP. The released Host Wire and Lycaon authority bridge remain compatibility paths during that migration.

Product goal

T4 Code presents one client experience across native local and managed execution profiles.

Architectural invariants

The following invariants apply to managed Hub sessions:

  1. PostgreSQL is authoritative for projects, sessions, commands, ownership epochs, approvals, progress events, and CI/CD associations.
  2. Kubernetes reconciles execution resources; Kubernetes objects are not the product event database.
  3. The T4 operator owns only T4 custom resources and their runtime Pods, Services, PVC claims, finalizers, and infrastructure conditions. It references but never installs or reconciles database, storage-provider, ingress, object-store, or observability operators.
  4. T4 creates every managed OMP session and therefore knows its legitimate owner from creation.
  5. Each accepted command is durable and idempotent before a runtime receives it.
  6. Each session has one current owner epoch. A stale runtime cannot claim commands or publish accepted state after ownership transfers.
  7. Runtime pods are disposable. Sessions, worktrees, commands, and user-visible history survive pod replacement.
  8. All runtimes for a project mount the same networked POSIX filesystem at the same absolute path.
  9. OMP owns repositories, Git worktrees, and its agent behavior inside the managed runtime. T4 does not invent a competing worktree model.
  10. T4 Code speaks only the versioned Hub protocol. It never depends on Kubernetes resources or direct pod connections.
  11. Development, single-node, and HA installations use the same T4 images, migrations, protocols, and portable charts. Optional stack profiles select tested dependency defaults without changing T4 semantics or transferring provider ownership to the T4 operator.
  12. No deployment is described as HA until it has at least three suitable failure domains and passes automated recovery checks.
  13. Product progress is durable application state. External telemetry supplements it but never replaces it.

T4 Local and Workstation Runner add these invariants:

  • T4 Local records accepted commands and client cursors durably in SQLite before dispatch.
  • Local and runner-backed sessions use native files and are explicitly nonportable.
  • A disconnected workstation session becomes unavailable; Hub never assigns it elsewhere.
  • Local files are never continuously synchronized into managed RWX storage.
  • Local Service is a single-machine authority, not a PostgreSQL replacement or cluster control plane.
  • Local Service, managed runtimes, and Workstation Runners reuse one official-OMP adapter and truthful capability manifest.

Managed system topology

This topology covers Personal Hub and HA Hub. T4 Local has no Kubernetes, PostgreSQL, network filesystem, object-store, or telemetry dependency; its client connects to native Local Service over loopback.

T4 Hub placement

In a single-node installation, one Linux machine may host T4 and compact dependency instances. In HA mode, T4 API and operator replicas are distributed independently from provider-managed database, storage, and telemetry replicas. Clients use one stable Tailnet service address and do not select a control-plane replica.

A Linux Hub machine may be a dedicated mini PC, home or office server, NAS-hosted VM, cloud VM, or Linux workstation. macOS and Windows run T4 Code and may run native T4 Local Service; macOS and Linux may additionally register as Workstation Runners. Managed T4 node implementations remain Linux-only, so a native Mac never needs a managed Linux VM for T4 Local.

Protocol boundaries

T4 Code consumes one state/action provider with Local and Hub implementations. Released @t4-code/host-wire remains the local compatibility contract during migration. The convergence target is a bounded capability-negotiated session surface shared by T4 Local Service and T4 Hub, without pretending that local endpoints provide cluster features.

Hub Wire

@t4-code/hub-wire is the external, versioned client contract. Its common surface covers:

  • pairing, authentication, devices, and capabilities;
  • projects, workspaces, repositories, and sessions;
  • prompt, steer, follow-up, cancellation, and attention responses;
  • bounded transcript, file, artifact, and image access;
  • durable progress cursors and reconnection;
  • endpoint profile, runtime capability, update, and compatibility status;
  • for Hub endpoints only, cluster health, backup, CI/CD, and deployment operations.

The current protocol's bounded decoding, branded identifiers, additive evolution, command idempotency, paging, and payload limits migrate where their semantics remain correct. Local filesystem authority stays inside Local Service; managed topology stays inside Hub. Flutter never receives Runtime Wire, Kubernetes resources, direct pod endpoints, Hub database access, or unowned absolute paths.

Runtime Wire

@t4-code/runtime-wire is the internal Hub-to-executor contract used by managed runtimes and Workstation Runners. The executor initiates the persistent connection, so pods and runners need no public inbound endpoint. It covers:

  • runtime or runner registration, endpoint profile, compatibility, and capabilities;
  • session identity, runtime image digest where applicable, and owner epoch;
  • durable command claiming and outcomes;
  • typed OMP prompt, steer, follow-up, cancellation, approval, and command operations;
  • ordered transcript and progress publication;
  • heartbeat, lease renewal, recovery checkpoint, completion, and failure.

A managed runtime may claim work or publish authoritative outcomes only for its current epoch. A Workstation Runner receives only runner-pinned nonportable sessions; losing its connection makes them unavailable rather than eligible for takeover. T4 Local Service can call the same adapter interface in-process and does not serialize an unnecessary local Runtime Wire hop.

OMP boundary

One T4-owned runtime adapter launches pinned official OMP through omp --mode rpc --session <path> and bounded NDJSON stdin/stdout. T4 Local Service, managed runtime pods, and Workstation Runners reuse this adapter. T4 clients never connect directly to OMP.

The adapter classifies every exposed operation as typed, verified headless, terminal-only, or unavailable. A recognized terminal-only command is rejected explicitly instead of becoming ordinary model prompt text. An optional separately versioned T4 plugin may use official extension APIs; missing generic capabilities should be proposed upstream. The Lycaon bridge remains temporary compatibility code, not a permanent private OMP product.

Current-code disposition

Retained and evolved

  • The shared Flutter client, adaptive GUI, secure Hub directory, session UX, transcript rendering, composer, attention inbox, developer surfaces, and lifecycle behavior.
  • Observable user workflows and deterministic fixtures that remain valid.
  • Bounded protocol and projection techniques.
  • Capability negotiation, command IDs, paging cursors, payload limits, and fail-closed decoding where their meanings survive.
  • The native Host lifecycle, pairing, local filesystem authority, terminal/file operations, and service packaging, evolved into T4 Local Service.
  • SQLite for the durable single-user local command ledger and client cursors.

Replaced after proof

  • Unbounded or ambiguous local OMP discovery and process probing.
  • Attaching externally owned sessions as the primary managed path.
  • Private OMP source imports, fork-only RPC assumptions, and the Lycaon bridge after parity.
  • Local JSONL compatibility projection where official durable RPC entries supersede it.
  • Competing-process takeover behavior that cannot prove one legitimate owner.

Replacement is gated by the official-adapter capability manifest and physical Local and Hub slices. T4 Local Service remains a supported production authority; only its private OMP integration is retired. Managed Hub does not absorb local lifecycle or filesystem ownership.

Session and command flow

The Local path uses the same command state machine with fewer hops:

T4 Code -> loopback Hub Wire -> T4 Local Service -> SQLite commit
        -> shared adapter -> OMP stdin -> OMP events -> SQLite outcome -> client cursor

A lost client connection cannot lose a committed command. Retrying the same command ID cannot execute it twice. A crash after dispatch but before provable OMP acceptance produces indeterminate, never automatic prompt replay. This applies to SQLite and PostgreSQL ledgers.

Session recovery

When a runtime pod fails:

  1. Its lease expires or the controller revokes it.
  2. T4 advances the owner epoch.
  3. The old runtime is fenced and terminated.
  4. The operator creates a replacement runtime using the same pinned image and worktree path.
  5. The replacement mounts the shared project filesystem.
  6. It restores the durable session checkpoint and resumes command/event cursors.
  7. The GUI moves from Recovering to Ready only after continuity is verified.

Shared project filesystem

All eligible execution nodes mount the same project filesystem at a stable path: 

/workspace/projects/<project-id>/
├── repo/
├── worktrees/
├── shared/
└── .t4/

OMP manages repositories and worktrees on this filesystem. Multiple sessions may use distinct OMP-managed worktrees concurrently while seeing the same project environment. Repository-global mutations must retain OMP/Git locking semantics and pass the storage conformance suite.

The live filesystem requires POSIX behavior, RWX mounting, atomic rename, exclusive file creation, symlinks, reliable permissions, recovery after node loss, snapshots, and acceptable metadata performance.

Managed deployments consume an administrator-selected Kubernetes StorageClass that provisions real RWX volumes. T4Workspace records the requested class and size; the T4 operator creates the PVC and mounts the bound claim into session pods. It does not install a CSI driver, create backend-specific CRs, select disks, or change storage topology.

The optional T4 Stack profile may install a vetted default storage chart for users who want a guided single-machine path. That provider retains its own operator, CRDs, RBAC, status, upgrades, and recovery procedures. An existing cluster may instead supply any implementation that passes the same contract.

Storage conformance covers real OMP worktree concurrency, POSIX locking, large repositories, package trees, pod and node failure, remount, snapshot, restore, stale-writer fencing, metadata performance, and git fsck integrity. A provider name or RWX declaration alone is not evidence.

Optional S3-compatible object storage is a separate contract for immutable artifacts, uploads, large transcript or log chunks, and backup objects. T4 receives an endpoint and credential reference; it does not expose provider-specific object-store CRs. The initial compact profile may keep bounded objects on RWX storage and omit S3 entirely.

Deployment profiles

T4 Code chooses an authority profile first. Managed installation guidance begins only for Hub profiles; local users never configure database, storage, or operator replicas.

T4 Local

T4 Local Service runs natively on the same macOS or Linux workstation as OMP and the project. Flutter connects over loopback by default. No Linux VM, Kubernetes cluster, PostgreSQL service, network filesystem, object store, or telemetry backend is required.

Benefits

  • Native filesystem, SDK, simulator, attached-device, and toolchain performance.
  • Lowest control-plane overhead and simplest offline behavior.
  • Optional paired Tailnet access while the workstation is awake.

Limitations

  • Sessions stop when the workstation sleeps, shuts down, or disconnects.
  • Local files and sessions are not portable or automatically failed over.
  • Backups remain the user's local storage responsibility.

Workstation Runner

A macOS or Linux machine registers outbound with an existing Hub and executes selected sessions against its native files and toolchain. Hub owns command history; the runner owns no control-plane database or scheduler.

Benefits

  • Hub collaboration and command durability with native Xcode, simulators, or local hardware.
  • No inbound runner port and no managed stack on the workstation.

Limitations

  • Every session is pinned to that runner and becomes unavailable when it disconnects.
  • No automatic takeover, local-to-managed-filesystem synchronization, or portable uncommitted state.

Personal Hub

One always-on Linux computer or VM runs the T4 components and managed runtimes. It may sit on a desk, on a LAN, or remotely; desktop and mobile clients connect through a stable Tailscale service. The optional T4 Stack chart supplies a tested compact path, while every database, storage, and telemetry dependency keeps an independent ownership boundary.

Benefits

  • Work continues while client devices are off.
  • Same Kubernetes APIs, protocol, operator, runtime image, and storage contracts as larger deployments.
  • Can expand into the intended HA topology.

Limitations

  • One failure domain: it is explicitly unprotected, not HA.
  • Full-stack resource floors must be measured; compact bounded retention is required.
  • External backup is required for meaningful disaster recovery.

Highly available Hub

At least three suitable Linux failure domains run Kubernetes quorum, distributed T4 replicas, a highly available PostgreSQL service, replicated provider-managed RWX storage, and schedulable runtime capacity.

Benefits

  • Survives a tested single-node failure and supports portable managed sessions.
  • Adds execution capacity, replicated state, and continuous team workloads.

Limitations

  • Requires at least three tested failure domains plus greater disk, memory, and network capacity.
  • Replication does not replace backups.

Two nodes may add capacity but are not labeled HA because they cannot preserve quorum after an arbitrary partition. No managed session moves across nodes until hard storage-writer fencing and owner-epoch recovery pass the failure matrix.

Existing-cluster installation

The portable charts support an existing conformant Kubernetes cluster, PostgreSQL service, RWX StorageClass, optional S3-compatible endpoint, ingress, and telemetry backend. The optional T4 Stack is a convenience distribution of the same T4 artifacts, not a second architecture and not authority for the referenced providers.

Guided managed installation

“On this computer” installs or connects to native T4 Local Service through the platform service manager and stops there. The following flow applies only when the user selects a Personal Hub or HA Hub.

First Hub

Stack-profile users do not need to run kubectl or place replicas manually. Existing-cluster administrators use Helm values and Secret references to bind T4 to services they already operate.

Cluster expansion

The GUI may expose Make highly available or Add node for a tested T4 Stack profile. T4 may enroll nodes, expand Kubernetes quorum, and add T4 replicas, but provider-specific database and storage expansion remains the responsibility of their own operators. The installer coordinates health gates without assuming that the T4 operator owns those topology changes.

The client endpoint, project identifiers, session history, worktree paths, and credentials remain stable during expansion.

Ordinary users choose among versioned, tested T4 Stack profiles. Existing-cluster administrators choose provider topology through the providers' own APIs, not through T4 custom resources.

Observability

T4 exposes bounded Prometheus-format metrics, structured logs, health endpoints, and OpenTelemetry signals. An administrator may connect Grafana, Prometheus, an LGTM deployment, or another compatible backend. The T4 operator does not install or reconcile those systems.

The portable chart may create narrowly scoped integration resources such as a ServiceMonitor or alert rules only when their CRDs already exist and the administrator enables them. The optional T4 Stack may offer a separately owned observability profile; it is not required for product correctness.

Observability failure cannot prevent session execution or alter durable product state. T4 Local requires no external telemetry service and uses bounded native diagnostics.

Performance and resource discipline

Ordinary prompt dispatch never waits on the Kubernetes API or telemetry. The hot path is one bounded client frame, one SQLite or PostgreSQL command-ledger transaction, one existing adapter or Runtime Wire stream, and one OMP stdin write. Kubernetes participates only in runtime lifecycle transitions such as create, stop, replace, resource change, and storage mount.

Instrument these timestamps from the first executable slices:

client send
service receive
command-ledger commit
runtime claim
OMP stdin write
first OMP event
event-ledger commit
client render

Measure the same scenario with bare official OMP, T4 Local, Personal Hub on one Linux machine, and a three-node HA Hub. Separate ledger durability, Runtime Wire, filesystem, provider, and presentation cost. Publish idle platform usage and incremental per-session CPU, memory, disk, inode, and network usage; profile floors come from those measurements rather than estimates.

T4 Local preserves native filesystem and toolchain performance. Managed benchmarks emphasize RWX metadata-heavy work: Git status, worktree creation, package trees, file watching, dependency installation, search, large builds, and concurrent OMP sessions. Personal Hub uses compact T4 replicas, conservative database buffers, bounded telemetry retention, node-local disposable caches, and suspended idle runtime pods. It adds no service mesh, Kafka, NATS, or per-runtime Tailnet sidecars without measured need.

Security boundaries

  • Hub APIs are Tailnet-only by default.
  • T4 Local Service binds loopback by default; optional Tailnet access requires pairing.
  • Device pairing produces scoped credentials and capabilities.
  • Node enrollment uses short-lived, single-use credentials and establishes durable mTLS identity.
  • Workstation Runners initiate outbound connections and receive runner-scoped authority.
  • Runtime pods receive only project- and session-scoped authority.
  • No runtime pod receives unrestricted Kubernetes API credentials.
  • Repository code is treated as untrusted: workloads require resource limits, network policy, controlled egress, and stronger sandboxing where the threat model requires it.
  • Secrets use workload identity or short-lived delivery and never enter transcripts, logs, images, support bundles, or repository files.
  • Support bundles are bounded and redacted by construction.
  • Every release includes signed OCI images, an SBOM, checksums, schema compatibility, and a supported-version manifest.

Delivery strategy

Work proceeds from the Flutter baseline already merged on main. The Local and Hub implementations converge on explicit common contracts; the current Host is evolved into Local Service instead of deleted or expanded into a disguised cluster control plane.

Backend and GUI order

T4 first proves the official OMP seam, freezes the common client, command, capability, and runtime contracts, then advances Local, Hub, shared runtime, managed infrastructure, and Flutter providers in parallel against executable fixtures. T4 Local reaches a physical slice before the Personal Hub slice; managed breadth follows recovery and restore evidence.

The existing Flutter T4ViewState and T4Actions boundary, adaptive shell, transcript rendering, composer, attention surfaces, and lifecycle UX remain the client seam. Native discovery, lifecycle, files, terminal operations, and OMP supervision stay encapsulated in T4 Local Service. Released Host Wire remains compatible until the common Local provider passes its physical slice.

Gate 0: stock OMP seam

Prove the highest-risk dependency before building the control plane. A small T4-owned harness launches an unmodified official OMP release through omp --mode rpc --session <path>, waits for its ready event, executes a prompt, steering or follow-up, approval response, cancellation, and a second prompt, then kills and respawns OMP against the same JSONL session on Linux. Transcript order and continuation must be reconstructed from durable session entries. A ready watermark may improve this later, but stock OMP 17.0.6 does not provide one.

This gate also resolves the dispatch crash window. Current stock OMP RPC identifiers correlate responses but do not durably deduplicate commands. T4 must either obtain a narrow upstream OMP capability that persists an opaque command key with the accepted session entry, or fail closed by marking a dispatched-but-unreconciled command indeterminate. It must never automatically replay an ambiguous prompt. Gate 0 does not pass by assuming host-side idempotency provides exactly-once OMP execution.

Implementation status (2026-07-21): the T4-owned supervisor now queries stock OMP's bounded get_available_commands RPC, normalizes typed and verified headless operations into the shared catalog, and supplements commands omitted from headless discovery with a manifest pinned to the reviewed OMP source revision. Local catalog responses expose those classifications. Recognized terminal-only prompts such as /plan fail with structured terminal_only details before any prompt is written to OMP stdin. The earlier lifecycle harness used the released t4code-17.0.5-appserver-10 Lycaon fallback artifact, despite its official-OMP name. The corrected harness separately pins unmodified official OMP v17.0.6 and its per-platform release hashes. On darwin-arm64 it now passes deterministic lifecycle, SIGKILL and same-session recovery, steering, follow-up, extension approval, and cancellation scenarios. Focused host and client tests prove that a crash after stdin dispatch produces an unknown outcome and never automatically replays the prompt. The checked Gate 0 snapshot records three missing stock seams: no ready transcript watermark, no live session_entry projection, and no durable command key. T4 therefore reconciles official OMP's JSONL and fails closed across ambiguity instead of pretending to provide exactly-once execution. Native linux-arm64 and linux-x64 CI jobs pass the same proof. Gate 0 is closed; packaged local and managed cutover remain a separate release gate.

Gate 1: contract freeze

Freeze the shared contracts that every later workstream consumes:

  • Common Local/Hub client state and action contracts plus Hub Wire and Runtime Wire fixtures, bounds, versions, endpoint profiles, capabilities, and error states.
  • The session state machine and the command transitions from pending through claim, dispatch, acceptance, completion, failure, and indeterminate outcome.
  • SQLite and PostgreSQL command-ledger semantics, payload hashing, durable outcomes, cursors, and crash-window behavior; PostgreSQL additionally owns outbox, owner epoch, and lease state.
  • Provider-neutral stale-writer fencing. A replacement cannot start until the old runtime is observed terminated or its write access is revoked and eviction is confirmed.
  • Immutable object upload, checksum, PostgreSQL manifest commit, orphan collection, and database-filesystem-object backup generation ordering for every enabled S3-compatible provider.
  • Stable project, session, command, cursor, runner, PVC, worktree, Tailnet service, OCI image, and compatibility identifiers.
  • Versioned Local, Personal Hub, HA Hub, and Workstation Runner schemas plus one explicit capability and failure-state table.

Gate 1 is a serial integration decision. Parallel implementation starts only after its fixtures and failure semantics are reviewable and executable.

Deliberate parallel work

After Gate 1, the following workstreams can proceed concurrently against the same fixtures:

Local systems Native service lifecycle, SQLite command ledger, local authority, official adapter integration, and loopback or optional Tailnet endpoint. Local crash-window, restart, filesystem, and nonportability conformance.
Hub core PostgreSQL migrations, command ledger, outbox, authentication, and Hub Wire API. Deterministic command and cursor conformance.
Shared runtime Official OMP supervisor and capability adapter, bounded NDJSON, Runtime Wire wrapper, transcript reconciliation, lease loss, and shutdown. Gate 0 scenarios repeated through both direct Local and Runtime Wire callers.
Operator T4 custom-resource reconciliation, pod identity, owner epochs, PVC claims and mounts, finalizers, and bounded replacement. No duplicate runtime across restart and lease-loss cases; no provider-specific CRs or cluster-wide dependency ownership.
Platform integration Portable Kubernetes and Helm APIs plus references to ingress, PostgreSQL, RWX storage, optional S3, and optional telemetry. One reproducible existing-cluster install and one optional single-node T4 Stack install.
Flutter Local and Hub providers projecting into retained state/action and adaptive GUI contracts. Common provider fixture conformance before physical Local and managed attachment.
Supply chain OCI builds, pinned digests, SBOM, signing, compatibility manifest, and release chart. The same artifacts run in CI and on the Linux node.

T4 chart semantics and provider contracts stabilize before the optional installer. Provider topology migrations remain owned and reported by their respective operators. A T4 Stack expansion coordinates these independent operations serially so failure attribution and rollback stay explicit.

Gate 2A: T4 Local slice

Evolve the current Host into native T4 Local Service with a durable SQLite ledger and the shared official-OMP adapter. Flutter creates a real local session, prompts, steers, reconnects after application restart, reads local files, executes one verified headless command, and rejects one recognized terminal-only command without sending it to the model.

Crash immediately after dispatch and confirm the command becomes indeterminate unless OMP proves durable acceptance. Close the workstation service and confirm no other machine claims the local session. This gate has no Kubernetes, PostgreSQL, network-filesystem, object-store, or telemetry dependency.

Gate 2B: Personal Hub substrate

Prove both supported installation forms: portable T4 charts against existing PostgreSQL, ingress, and RWX services; and the optional T4 Stack on one real Linux node. Both forms deploy the same Hub, operator, and runtime artifacts. Docker Desktop, k3d, and a separate Docker Compose architecture are not required.

Before running OMP, mount one conformant RWX project from multiple disposable pods, reboot the node, remount it, and verify filesystem semantics. Database, filesystem, optional object-store, and telemetry health remain distinguishable; external-service impairment cannot grant the T4 operator provider authority or corrupt product commands.

Gate 3: Personal Hub physical-client slice

Prove one complete path with real components:

  1. T4 Code connects over Tailscale to the stable Personal Hub service.
  2. T4 Hub persists a project, session, and prompt in PostgreSQL.
  3. T4 Operator starts a real T4 Session Runtime with the correct administrator-selected RWX project mounted.
  4. The runtime launches pinned official OMP and reconciles its durable entry watermark or fails closed where stock OMP cannot provide one.
  5. OMP creates or uses its worktree and changes a file on the RWX volume.
  6. Durable progress and transcript events reach the existing Flutter GUI projection.
  7. A desktop client and a physical mobile client observe the same authoritative session.
  8. Stage timestamps isolate client, PostgreSQL, Runtime Wire, OMP, filesystem, and presentation overhead; idle and active resource floors are recorded.

A deterministic protocol client runs this path before Flutter to localize server defects, but the phase does not pass until physical Flutter clients complete it. Managed backend work does not drift into enabled capabilities ahead of this slice.

Gate 4: recovery and restore

Exercise every command dispatch crash window, OMP child failure, runtime pod deletion, Hub API restart, operator leader restart, Tailnet interruption, and full single-node reboot. A stale runtime must not modify the shared worktree after replacement, an ambiguous prompt must not be replayed, transcript cursors must converge, and Git integrity must remain valid.

Create a coordinated PostgreSQL, shared-filesystem, and optional object-store backup generation on an independent destination, wipe the test installation, and restore it onto a clean host. Backup creation without a successful bare-host restore is not evidence.

Workstation Runner proof

Register one macOS or Linux runner with the proven Hub, pin a session to it, use its local toolchain and filesystem through the shared adapter, disconnect it, and confirm the session becomes unavailable without reassignment or prompt replay. Reconnect the same runner and reconcile against the same local OMP session and files.

Gate 5: guided installer

Turn the exact managed automation used by Gates 2B through 4 into the signed T4 Node bootstrap and Flutter setup flow. Installation, health checks, support collection, update, failed-update stop, rollback, and recovery must be repeatable from a clean supported Linux image without making an ordinary stack-profile user administer Kubernetes or its independently owned providers.

Gate 6: HA expansion

Add two validated Linux nodes and perform changes serially: expand Kubernetes quorum, request provider-owned database and storage expansion, add Hub and T4 operator replicas, then validate the stable Tailnet endpoint. Every provider must report its own healthy state, and the T4 operator must remain unable to mutate provider topology. Remove one node and prove continued operation before applying the HA label.

Product breadth and CI/CD

After Gate 4, approvals, questions, steering, cancellation, files, terminals, artifacts, and remaining managed workflows proceed as vertical backend-plus-GUI slices. GitHub App, checks, self-hosted runner, and deployment progress can proceed in parallel once command durability, event ordering, and workspace recovery are stable. No CI/CD integration may bypass the command ledger or product event model.

First Linux test node

The offered Tailnet Linux box is the correct first external environment. It is disposable test infrastructure, not production data, and should provide:

  • A supported Linux distribution, kernel, cgroup mode, and OCI image architecture.
  • Root or controlled sudo access, a stable hostname, synchronized time, and reboot persistence.
  • Storage capacity and any provider-specific devices required by the selected optional T4 Stack profile, explicitly approved and containing no data to preserve.
  • Measured CPU, memory, root-disk, inode, and provider capacity headroom for one OMP session plus the compact platform; final profile floors come from this evidence rather than guesses.
  • Tailnet enrollment, intended ACL/tag policy, working MTU, OCI and source-provider egress, and no required public inbound port.
  • An independent off-node backup destination for destructive restore testing.
  • Disabled sleep and an understood power-loss/restart policy.

Hostnames, Tailnet addresses, enrollment credentials, provider tokens, and private repository data remain runtime configuration and never enter the repository or documentation.

Reliability and release gates

A milestone is not complete from compilation, a running local service, or a healthy Kubernetes dashboard. It must prove observable behavior and failure semantics for the advertised profile.

Command and ownership

  • A command ID is committed once and is never knowingly dispatched twice.
  • An ambiguous post-dispatch outcome becomes indeterminate, never auto-replayed.
  • A client can recover every durable outcome after transport loss.
  • A stale managed owner cannot claim commands or publish authoritative outcomes.
  • Managed owner loss produces one fenced replacement; local or runner loss produces unavailability.

Workspace

  • Concurrent real OMP worktrees preserve repository integrity.
  • Managed pod and node loss preserve work only after the selected storage provider passes fencing and recovery proof.
  • Local and runner files remain on their named machine and are never claimed as portable.
  • Expansion from one node to three does not change managed paths or project identity.
  • Snapshot restoration reproduces database and filesystem generations consistently.
  • Representative destructive tests finish with successful Git integrity checks.

Managed control plane

  • API replica loss does not interrupt durable commands.
  • Operator leader loss does not duplicate resources.
  • PostgreSQL primary loss preserves committed state.
  • Tailnet reconnect resumes from cursors rather than replaying unbounded history.
  • Optional object-store or telemetry impairment degrades only explicit dependent features without corrupting product state.

Client

  • Physical desktop clients create, operate, restart, and inspect a real T4 Local session.
  • Physical desktop and mobile clients create, operate, recover, and inspect one managed session.
  • Compact and wide layouts show creating, ready, running, waiting, unavailable, recovering, indeterminate, failed, and completed states truthfully.
  • Background/resume and network switching preserve command and transcript continuity.

Installation and lifecycle

  • A non-expert can install T4 Local Service on macOS or Linux without a VM or managed stack.
  • A Workstation Runner installs separately and cannot acquire Hub control-plane authority.
  • A non-expert can install a single remote Linux Hub without Kubernetes knowledge.
  • Adding two validated nodes converts it to HA only after replication and failure checks pass.
  • Upgrade runs preflight, database backup, filesystem snapshot, signed-image verification, migration, canary, and health gates.
  • Failed upgrades stop safely and expose a tested recovery or rollback path.
  • Backup restoration is exercised, not inferred from successful backup creation.

Out of scope

  • Native macOS or Windows Hub implementations.
  • Docker or Docker Desktop as a user prerequisite.
  • Attaching arbitrary externally owned desktop OMP sessions to the managed control plane.
  • Automatic takeover of Local or Workstation Runner sessions.
  • Continuous bidirectional synchronization between local files and the managed shared filesystem.
  • Installing or lifecycle-managing database, filesystem, object-store, ingress, or observability providers from the T4 operator.
  • Active-active cross-cluster filesystem failover.
  • Making an observability backend the user-facing product event database.
  • Retaining the private Lycaon OMP bridge after official-adapter parity is proven.

Open decisions requiring proof

  • Whether the remaining official OMP lifecycle, recovery, and durable acceptance behaviors require narrow upstream additions. Capability discovery and terminal-only dispatch classification use the existing official RPC seam.
  • Measured overhead and resource floors for T4 Local, Personal Hub, and HA Hub profiles.
  • Tailnet hosted-service plan requirements and whether Headscale meets the supported deployment contract.
  • Packaging and platform capability details for macOS, Linux, Windows, and GPU runners.
  • Retention defaults for product events, transcripts, artifacts, backups, and optional external telemetry.
