Status and decision

This hand-maintained document supersedes the prior T4 Code local and managed architecture and is the canonical architecture decision for the managed T4 product. It defines a conventional Kubernetes Operator and a separately deployable, horizontally scalable, versioned T4 API. It is a target contract, not a claim that every described component is already implemented. The implementation baseline and the new work are called out explicitly below.

The product boundary is authenticated HTTPS T4 API v1. External clients do not call the Kubernetes controller, do not receive Kubernetes credentials, do not mutate custom resources, and do not speak a private runtime protocol. PostgreSQL is authoritative for accepted intent and product events. Kubernetes custom resources are a durable projection of that committed intent. The controller reconciles that projection into Kubernetes resources; it never serves product clients or stores product events.

Scope and terms

TermMeaning
T4 API The authenticated HTTPS product service implementing T4 API v1 from executable OpenAPI. It owns request validation, authorization, idempotency, intent and event persistence, watches, and public capability discovery.
Gateway The deployable edge/API module. It terminates or receives verified identity, applies scope policy, commits intent to PostgreSQL, then projects committed intent into namespaced T4 custom resources.
Controller The Kubernetes reconciliation process. It watches T4 custom resources and owned children, converges desired state, and reports bounded infrastructure status through Kubernetes status.
Runtime provider The typed internal boundary that starts, supervises, fences, and communicates with a pinned supported OMP runtime. Product modules do not depend on OMP wire details directly.
Terzo The T4 client. Terzo consumes the generated public SDK and never imports controller packages or Kubernetes resource clients.
ACP adapter An optional local stdio compatibility process for standard editor clients. It speaks Agent Client Protocol 1.2.1 using JSON-RPC 2.0 NDJSON on local stdio and translates to T4 API v1 over HTTPS. ACP is not the remote network API.

Implementation baseline

Already-merged foundations

The following are present foundations and must be evolved rather than described as future inventions:

  • Namespaced cluster.t4.dev/v1alpha1 T4ClusterHost, T4Workspace, and T4Session custom resources with structural schemas.
  • Status conditions and observedGeneration, deletion finalizers, owner references, leader election, bounded requeues, and namespace-scoped controller RBAC.
  • Helm packaging, controller and server images, gateway foundations, network policies, health endpoints, metrics, and observability foundations.
  • Durable-command concepts, owner epochs and fencing, bounded protocol frames, cursor recovery, disposable runtimes, and shared-filesystem eligibility checks.

New work required by this decision

  • Executable T4 API v1 OpenAPI, generated public SDKs, conformance fixtures, version and capability discovery, and stable error schemas.
  • PostgreSQL migrations for durable intent, idempotency, ordered events, commands, cursors, and a projection/outbox worker. Existing in-memory idempotency is not sufficient for the final product.
  • A clean controller-only contract, the typed OMP runtime provider, Terzo migration to the public SDK, and the optional local ACP adapter.
  • Signed multi-architecture OCI artifacts, SBOM and provenance, compatibility preflight, supported migration and rollback procedures, and physical remote-client and HA evidence.

Architectural invariants

  1. PostgreSQL authority. A product mutation is accepted only when its intent and idempotency record commit in PostgreSQL. Kubernetes is a durable desired-state projection, not the product event ledger.
  2. Operator-only reconciliation. The controller has no public listener. It reconciles custom resources and Kubernetes children and writes status; it does not authenticate product users, serve watches, or retain product commands or events.
  3. Commit before projection. The gateway may create or patch a custom resource only after the corresponding database transaction commits. A projection can be retried from the durable outbox without accepting the intent again.
  4. Stable ownership. Every runtime owner carries a monotonically increasing owner epoch. Dispatch includes that epoch; stale owners are fenced before replacement owners may act.
  5. Disposable runtime. Runtime pods and local child processes are replaceable compute. Durable intent, events, cursors, and workspace data survive their loss.
  6. Reconnectable observation. Watches use opaque durable cursors. Reconnect resumes after the last acknowledged cursor; cancel is explicit and bounded; expired cursors return a typed resync requirement.
  7. Shared-filesystem safety. A managed runtime starts only when the selected workspace claim is bound and satisfies the reviewed ReadWriteMany contract. Labels do not make an unsafe backend safe.
  8. Truthful capabilities. Discovery distinguishes supported, enabled, authorized, and currently available behavior. An unavailable dependency removes or disables a capability; it never causes a successful placeholder response.
  9. Bounded work. Request bodies, frames, queues, retries, watches, log payloads, and child-process output are bounded. Backoff is capped and jittered; terminal conditions do not busy-loop.
  10. No implicit exactly-once claim. Database acceptance is idempotent. Runtime side effects use command identifiers, state transitions, owner fencing, and reconciliation to prevent known duplicate dispatch; uncertain crash windows remain visible and are recovered according to the command contract.

Kubernetes reconciliation contract

Reconciliation follows ordinary Kubernetes level-based semantics. It is idempotent and derives work from current desired and observed state, not from an assumption that every watch event is delivered exactly once.

  1. Watches on a primary T4 resource and its owned secondary resources enqueue a namespaced key. The active leader fetches the latest object; a missing object completes without error.
  2. The reconciler validates the structural spec and referenced host, runtime profile, storage eligibility, and policy. Invalid or terminal inputs produce a stable condition and do not use rapid retries.
  3. For a live object, the controller adds the documented finalizer before creating dependent state. It computes desired children and applies or patches them with deterministic names, labels, field ownership, and controller owner references where lifecycle ownership is intended.
  4. For deletion, the controller performs retention-aware cleanup. A T4Session finalizer waits for its Pod and Service to be absent. A T4Workspace with delete retention waits for its PVC to disappear; retain retention removes the owning reference and records retention before finalizer removal.
  5. Status is patched separately from spec. Conditions have stable type, status, reason, and concise message fields. The controller advances status.observedGeneration only after evaluating that generation; readiness is never inferred from generation alone.
  6. Optimistic conflicts and transient API failures retry through the work queue. Dependency polling uses bounded, capped, jittered requeues. Successful convergence relies on watches rather than periodic churn.

The controller may read projection identifiers placed on a CR by the gateway and report Kubernetes status correlated to that projection. It must not reach into the product ledger to accept commands or manufacture product events. The API converts correlated Kubernetes status into public events under its own transaction and schema rules.

CRD group, version, storage, and conversion lifecycle

cluster.t4.dev/v1alpha1 remains the current served and storage API for the namespaced T4ClusterHost, T4Workspace, and T4Session resources. Current changes are additive and structural: new fields are optional or have safe schema defaults, unknown fields are not used as an extension mechanism, and existing persisted objects must continue to validate.

  • Helm installs workloads but is not trusted to upgrade CRDs. Release automation applies reviewed CRD manifests server-side before workload rollout and waits for each CRD to be Established. A validation failure stops the upgrade; automation never force-replaces a CRD or deletes a stored version.
  • A v1beta1 proposal requires separate schema, compatibility, defaulting, conversion, round-trip, and downgrade review. Until that review passes, v1alpha1 remains both served and storage.
  • When conversion is approved, deploy and prove a highly available conversion webhook before making the new version served. Conversion must preserve hub data and round-trip every fixture between the old and new versions without semantic loss.
  • Change the storage version only after all supported controllers and gateways can read both versions. Run a storage migration, verify status.storedVersions and object counts, and retain the old served version through the documented rollback window.
  • Remove an old served or stored version only in a later incompatible lifecycle step after backups, migration evidence, compatibility-window expiry, and an explicit operator action. Rollback never requires an older binary to read a schema it does not support.

T4 API v1 and gateway contract

The remote contract is an executable OpenAPI document served and versioned with T4 API v1. The checked-in specification is the source for generated SDKs, request/response validators, examples, fixtures, and compatibility checks. Hand-written client DTOs are prohibited where generated public types exist.

Required v1 surface

  • Version and capability discovery, including server build, API versions, feature identifiers, bounds, enablement, authorization, availability, and deprecation metadata.
  • Workspace and session intent creation, retrieval, listing, mutation, deletion, and cancellation with scoped principals and stable resource identifiers.
  • Command submission with a required idempotency key and typed accepted, rejected, conflict, unavailable, and indeterminate states.
  • Snapshot reads and ordered watch streams with opaque cursors, reconnect-after-cursor, heartbeat, explicit cancellation, bounded batches and frames, and typed cursor expiry requiring resync.
  • Stable errors carrying a machine code, safe message, request ID, retry classification, and field violations without exposing internals.

Every mutation authenticates a principal, authorizes tenant, namespace, resource, and action scope, validates advertised bounds, and records an audit correlation ID. The same principal and idempotency key with the same canonical request returns the original result. Reuse with a different request is a typed conflict. Keys are isolated by principal and operation scope and expire only according to a documented retention policy.

API and controller compatibility is explicit. Startup and readiness fail closed when database schema, CRD served/storage versions, or runtime provider versions are outside the supported matrix. Public capability discovery exposes only behavior that the current compatible deployment can actually complete.

Durable intent, event, and command ledger

Mutation acceptance ordering

  1. Authenticate, authorize, bound, and canonically validate the request.
  2. Begin a PostgreSQL transaction and acquire the required aggregate or command serialization lock.
  3. Insert or resolve the unique scoped idempotency record. A matching committed record returns its stored response; a mismatched fingerprint fails with conflict.
  4. Append immutable accepted intent and product event rows, allocate the aggregate sequence and durable cursor, update the materialized current state, and insert a projection/outbox row in the same transaction.
  5. Commit PostgreSQL before reporting acceptance.
  6. After commit, a retryable worker creates or patches the corresponding custom resource with stable intent and projection identifiers. It then records projection progress without rewriting accepted history.

Per-aggregate sequence numbers and a globally comparable opaque cursor order committed events. A watch transaction establishes a snapshot boundary and cursor so that events cannot be silently lost between list and watch. Clients acknowledge only cursors they have applied. Event retention may compact old payloads only after retaining a snapshot and a deterministic expired-cursor response.

Command state and ownership

A command has a durable identifier, idempotency scope, target session, owner epoch, request fingerprint, timestamps, and an explicit state such as accepted, projected, dispatching, running, succeeded, failed, cancelled, or indeterminate. Transitions are conditional and append an event. A worker may dispatch only after acquiring the session ownership lease and recording the current epoch. Runtime acknowledgements must echo command ID and epoch; stale epochs are rejected.

Cancellation is intent, not deletion of history. The ledger records the request, the provider attempts bounded cancellation, and the final event truthfully distinguishes cancelled-before-start, cancellation-confirmed, already-completed, and cancellation-uncertain outcomes.

Crash-window behavior

WindowRequired recoveryObservable proof
Before database commit No intent is accepted and no CR may be mutated. Retry with the same key can create exactly one accepted record.
After commit, before HTTP response The idempotency record returns the committed result; projection proceeds from the durable outbox. Client retry returns the original resource and cursor.
After commit, before CR projection A new worker safely retries the stable create or patch. One logical projection exists and reaches correlated status.
After dispatch record, before runtime acknowledgement Recovery queries the provider by command ID when supported; it never silently reports success or knowingly dispatches through a stale epoch. Unresolvable outcomes become indeterminate. The event stream exposes the recovery or indeterminate state.
After runtime completion, before outcome commit Recovery obtains the durable provider result when available and conditionally commits one terminal transition; otherwise it reports indeterminate rather than inventing an outcome. At most one terminal ledger transition is committed.
Owner loss during execution Lease expiry advances the owner epoch, fences and terminates the old runtime, then permits a replacement to recover durable session state. Old-epoch heartbeats, events, and outcomes are rejected.

Runtime provider and OMP boundary

OMP is behind one T4-owned typed runtime-provider interface. The provider pins a tested official OMP release and one explicitly supported local transport: the supported OMP RPC seam or ACP 1.2.1 where that operation is available and verified. Transport selection is an implementation detail and is never exposed as the remote product protocol.

  • Launch uses an explicit executable, arguments, environment allowlist, working directory, protocol version, maximum frame size, startup deadline, and shutdown deadline. There is no unbounded process probing.
  • The provider maps sessions, prompts, questions, approvals, steering, cancellation, commands, files, terminals, and runtime events into typed T4 operations only when the pinned OMP seam proves them.
  • Capability discovery combines the pinned compatibility manifest with live bounded discovery. Manifest-only behavior remains unavailable until an executable probe passes; unknown operations fail closed.
  • The provider enforces command IDs and owner epochs on every operation, bounds stdout and stderr, handles malformed frames as typed failures, and never treats process exit as proof of command success.
  • Runtime hosts use a stable workspace mount, contain no authoritative product state, and can be destroyed after events and outcomes are durably acknowledged.

Terzo and ACP compatibility flows

Terzo remote flow

  1. Terzo authenticates and selects a T4 API endpoint.
  2. It calls version and capability discovery through the generated SDK and rejects incompatible major versions or required unavailable features.
  3. It creates or mutates intent with a persisted idempotency key. The API commits intent before a CR projection and returns the accepted resource and cursor.
  4. Terzo obtains a snapshot and watches after its cursor. On disconnect it reconnects with the last applied cursor; on cursor expiry it performs a typed resync. User cancellation is another idempotent API mutation.

Workspace, recovery, and backup

Shared filesystem

Managed execution nodes mount the selected project claim at a stable configured path. A workspace becomes runtime-eligible only after the PVC is Bound and the selected StorageClass is explicitly reviewed for the required ReadWriteMany semantics. The controller requests ReadWriteMany and fails closed for absent, ambiguous, or ineligible classes. Concurrent work uses isolated Git worktrees or an equivalently reviewed workspace strategy; shared writable access does not permit concurrent mutation of one worktree.

Session recovery

  1. The failed or unreachable runtime's lease expires or is revoked.
  2. The durable owner epoch advances in a conditional transaction.
  3. Old credentials and epoch traffic are rejected and the old runtime is terminated.
  4. A replacement runtime mounts the same eligible workspace and restores only acknowledged durable session/provider state.
  5. Commands in uncertain windows follow the crash-window table; clients resume from their durable cursors and see the recovery transition.

Backup and restore

  • PostgreSQL backups include schema migrations, intents, idempotency, events, commands, cursors, ownership, and projection state. Point-in-time recovery configuration and restore credentials are tested, not merely documented.
  • Workspace and object-store backups use provider snapshots or a quiesced procedure with an explicit consistency boundary correlated to a database recovery point. Kubernetes manifests alone are not a product backup.
  • Restore starts into isolated networking, restores and verifies PostgreSQL first, restores workspace/object data to the matching point, applies compatible CRDs, deploys compatible API and controller images, then replays outstanding projections.
  • Restore rotates runtime and service credentials, advances owner epochs, and reconciles stale projected resources before accepting mutations. Evidence includes resource counts, cursor continuity, checksums or provider verification, and one recovered end-to-end session.

Security boundaries

  • The only external product boundary is authenticated HTTPS T4 API v1, Tailnet-only or equivalently private by default. TLS termination, forwarded identity, origin, and proxy trust are explicit and bounded.
  • Human, device, service, projection-worker, controller, and runtime principals are distinct. Authorization is deny-by-default and scoped by tenant, namespace, resource, verb, and capability. User credentials are never mounted into controller or runtime pods.
  • API and controller use separate least-privilege ServiceAccounts. The API can perform only reviewed CR reads and projections; the controller can reconcile only reviewed T4 and child resources. Neither receives cluster-admin.
  • Runtime-to-service calls use short-lived, audience-bound workload identity or mutually authenticated transport. Epoch and session scope are checked in addition to identity. Secrets are referenced, not copied into CR specs, status, events, logs, or public errors.
  • Default-deny NetworkPolicies allow only required DNS, Kubernetes API, database, ingress, observability, runtime, and dependency paths. The controller has no ingress from public clients.
  • The local ACP adapter communicates with its parent through inherited stdio, does not listen publicly, validates frame bounds, redacts tokens, and stores long-lived user secrets only through platform secure storage.
  • Audit events cover authentication decisions, scoped mutations, idempotency conflicts, projection, privileged runtime actions, cancellation, migration, restore, and administrative policy changes. Payloads and logs are redacted and retention is explicit.
  • OCI signatures, SBOMs, provenance, pinned digests, admission policy, and preflight verification form the supply-chain boundary. Capability flags do not bypass authentication or authorization.

Distribution, migration, and lifecycle

Distribution follows the existing Helm and image conventions. The final release adds signed multi-architecture OCI images and charts, per-artifact SBOMs, build provenance, immutable digests, a compatibility manifest, and operator-facing preflight and rollback instructions. These are required work, not current claims.

Upgrade order

  1. Back up PostgreSQL and workspace/object data; record deployed versions, digests, CRD stored versions, and database migration level.
  2. Preflight Kubernetes, storage, PostgreSQL, ingress, identity, CRD, conversion-webhook, API/controller/runtime, architecture, and free-space compatibility. Verify signatures and provenance before pulling into the trusted runtime.
  3. Apply additive CRD changes outside Helm and wait for establishment. Do not change storage version unless the reviewed conversion lifecycle is being executed.
  4. Run backward-compatible, expand-first PostgreSQL migrations. New API replicas remain unready until the migration and CRD compatibility checks pass.
  5. Roll out API replicas, projection workers, controller replicas, then runtime images using disruption budgets and readiness. Enable new public capabilities only after the full path is healthy.
  6. Prove an idempotent mutation, projection, reconciliation, runtime event, cursor reconnect, and cancellation before declaring success. Contracting database or CRD migrations occur only in a later release after the rollback window.

Rollback rules

Rollback pins the prior signed image and chart digests and is supported only while database and CRD changes remain backward-readable. Traffic is drained before API rollback; projection workers are paused if their schema is incompatible. CRDs are never force-replaced or blindly downgraded. Failed conversion or storage migration stops forward rollout and restores from the verified backup when in-place compatibility cannot be proven. Feature capability remains disabled until all components again agree.

Observability and truthful capabilities

API, projection worker, controller, and runtime provider expose separate health, readiness, bounded Prometheus metrics, structured logs, and OpenTelemetry traces with request, intent, projection, resource, command, and cursor correlation identifiers. High-cardinality user, prompt, path, and command text are not metric labels.

  • API readiness covers PostgreSQL migration compatibility, CRD discovery, required identity configuration, and projection-worker health; it does not claim every optional capability is available.
  • Controller readiness covers Kubernetes access, informer synchronization, leader-election participation, and compatible served CRDs. Leader status is observable separately from process health.
  • Required metrics include accepted and rejected intents, idempotency replay/conflict, transaction latency, outbox age and failures, watch lag and resync, reconcile result and duration, finalizer age, condition age, owner-epoch changes, stale-epoch rejection, runtime restarts, command state age, backup age, and restore verification.
  • Alerts are actionable and bounded: stuck outbox, unavailable API quorum, controller leader absence, reconcile error budget, finalizer stall, command indeterminate growth, database recovery risk, cursor retention pressure, and backup verification failure.
  • Capability discovery derives from compatible versions, configured policy, principal scope, dependency health, and verified runtime probes. Build-time support alone is not reported as runtime availability.

Release proof and open decisions

Compilation, a healthy dashboard, mock transport, or an in-process client is not release proof. A release candidate must satisfy OP-10 using signed artifacts and real failure boundaries. Evidence records exact versions, digests, topology, timestamps, injected fault, expected invariant, observed events and cursors, metrics, logs, and recovery result.

The following remain decisions requiring executable proof rather than promises: which pinned official OMP operations use its RPC seam versus ACP; the initial cursor and idempotency retention windows; the first supported PostgreSQL, Kubernetes, ingress, identity, and RWX-provider matrix; and the exact conditions for proposing v1beta1. These choices may narrow advertised capability or supported platforms, but they may not weaken commit-before-projection, fencing, security, or recovery invariants.
